# 07 — i18n

> **Owns:** the localization mechanism — locale model, label-catalog format and loading, message library, string discipline and its enforcement, number/currency/date formatting, missing-key behavior, translation workflow. The v0 label strings themselves live in `ui-labels.md` (the catalog seed). Envelope facts live in `05-operation-log.md`; module manifest shape in `04-module-contract.md`.
> **Change control:** change this doc first, then the code.

## 1. Locale model

| Locale code | Meaning | Status in v0 | Intl locale tag used for formatting |
| ----------- | ------- | ------------ | ----------------------------------- |
| `id` | Bahasa Indonesia | **default**; source language — every string is written in `id` first | `id-ID` |
| `en` | English | selectable in-app toggle | `en-GB` (day-first dates; see §5) |
| `zh` | Chinese | **scaffolded, not translated** — present in the `Locale` type and fallback chain; no catalog files; NOT selectable in the v0 UI. Translation is a V2 agent task (PRD-012 FR-1159, PRD-004 §3.12). | `zh-CN` (reserved) |

- `Locale = 'id' | 'en' | 'zh'` and the selectable-locale list (`SELECTABLE_LOCALES`) are the **shared locale vocabulary**, declared **once** in `@bolusi/schemas` (`src/locale.ts`) — the zod-only, platform-free package that both `@bolusi/i18n` (which re-exports them and adds the Intl-tag map + i18next wiring) and `@bolusi/core` (the `platform.setLocale` / `platform.user_locale_changed` enums, §1.1) import. It lives in `schemas`, not `i18n`, because `@bolusi/core` may not import `i18n` (it pulls a runtime library; 08 §3.3), and a shared home is the only way the in-app toggle and the op enum cannot drift (CLAUDE.md §2.8; task 77). Adding a locale is a change to this doc first, then `@bolusi/schemas`.
- Locale codes are BCP 47 primary language subtags. Nothing else (no `id-ID` as a *locale code* — the region tag is a formatting concern only, §5).

### 1.1 Per-user preference — synced as an operation

Nothing is stored as current state (00-product-overview); the locale preference is no exception. Changing language emits a **platform-module** operation through the command runtime (04-module-contract §5):

| Field | Value |
| ----- | ----- |
| `type` | `platform.user_locale_changed` — registered in 01-domain-model §6's op catalog |
| `entityType` / `entityId` | `user_pref` / the acting `userId` (per the owning registry row, 01-domain-model §6) |
| `schemaVersion` | 1 |
| `payload` | `z.object({ locale: z.enum(SELECTABLE_LOCALES) }).strict()` — the enum is fed by the shared `SELECTABLE_LOCALES` (§1), so it accepts exactly what the toggle offers (`['id','en']` in v0); `zh` becomes legal here in V2 by joining that one list |
| `reversal` | "Reversed by a subsequent `platform.user_locale_changed` carrying the previous locale." |
| Command | `platform.setLocale` — any active user may set their own locale; permission `platform.set_locale` (registered in 02-permissions §11, granted to every role) |

Projection: a `user_prefs` row keyed by `userId` with a `locale` column (entity-scoped per 04 §4.1; registered in 01-domain-model §7's projection list, DDL in 10-db-schema for both engines with an RLS row). On PIN switch, the UI locale becomes the incoming user's projected preference (default `id` if none).

### 1.2 Device-level pre-login locale

Before any user is present (enrollment screen, user switcher, PIN pad), the UI renders in the **device locale**: the locale of the last active user, persisted in plain local storage. Default `id`. This is unsigned UI state, not business truth — it is NOT an operation and never syncs.

## 2. Library: i18next + react-i18next + i18next-icu

The stack-research snapshot (2026-07-14) did not cover i18n libraries; this section is the owning decision.

| Option | Verdict |
| ------ | ------- |
| **i18next + react-i18next + i18next-icu** | **Chosen.** Boring default with the largest ecosystem; runtime-universal (same core runs in the Hono server for push composition, §8); react-i18next is the standard RN binding; `i18next-icu` gives full ICU MessageFormat syntax in catalog values; no compile-step magic; first-class TS key typing. |
| react-intl (FormatJS) | Rejected: component-first API couples strings to React; no clean server-side reuse of the same instance; heavier per-render ceremony for an app that is mostly `t(key)`. |
| lingui | Rejected: macro/compile pipeline (Babel/SWC plugin) adds build complexity on Expo/Metro for no v0 benefit; smaller ecosystem. |

- **Version pinning happens at implementation** — the implementing agent shall verify current docs (Context7) for `i18next`, `react-i18next`, and `i18next-icu`, then pin exact versions in the lockfile. Do not copy version numbers from training data.
- **Hermes caveat (must verify on device):** Hermes ships a *subset* of `Intl`. `Intl.NumberFormat`, `Intl.DateTimeFormat` are available on Android; `Intl.PluralRules` and `Intl.RelativeTimeFormat` have historically been missing. `i18next-icu`'s plural selection depends on `Intl.PluralRules`. Contingency (decided now so nobody improvises): if the §5.4 device test fails, ship `@formatjs/intl-pluralrules` polyfill — and nothing else from FormatJS. `Intl.RelativeTimeFormat` is never used (§5.3).
- i18next config (normative):

```ts
i18next.use(ICU).init({
  lng: deviceLocale,            // §1.2, then per-user after switch (§1.1)
  fallbackLng: 'id',            // §6
  resources,                    // merged catalogs, §3.3 — bundled, never fetched
  interpolation: { escapeValue: false },   // React escapes; ICU output is text
  returnNull: false, returnEmptyString: false,
  saveMissing: true, missingKeyHandler,    // §6
  parseMissingKeyHandler,                  // §6
})
```

## 3. Label catalog format

### 3.1 Key grammar

```
<namespace>.<screen-or-area>.<label>        e.g. auth.pin.title, notes.list.empty
```

- Segments: lowercase camelCase, dot-separated, ≥ 3 segments. English internal names only.
- **The grammar binds every key, wherever it is written** — `ui-labels.md` rows, reserved-namespace catalogs, and module-owned catalogs alike. A key that is awkward to name under this grammar is a signal that its screen/area is unclear, not grounds for a 2-segment exception: name the area, or leave the key out until the screen exists (task 30 — `auth.switchStore` took the latter route; see roadmap R22).
- **Exception — derived error keys** (§4.3): the final segment of `core.errors.*` and `core.rejection.*` is the SCREAMING_SNAKE code verbatim (`core.rejection.CHAIN_BROKEN`), so the key is mechanically derivable from the code — no hand-written mapping table may exist.
- **Exception — derived registry keys** (`permission.*`, `role.*`): permission labels use the 4-segment keys `permission.<module>.<action>.name` and `permission.<module>.<action>.description`, where `<module>.<action>` is the permission id from 02-permissions §11 verbatim (snake_case segments legal here); role labels use `role.<roleKey>.name`, where `roleKey` is exactly one of the three seeded roles `main_owner | store_owner | staff` (01-domain-model §4.2). Both key families are mechanically derived from their registries — no hand-written mapping table may exist.
- **Namespace = first segment.** Reserved namespaces (platform-owned): `core`, `auth`, `sync`, `conflict`, `media`, `push`, `permission`, `role`. The `permission` and `role` namespaces contain only the derived registry keys above; `push` holds the server-composed notification templates (api/04-push.md, §8). Every `defineModule` module owns exactly the namespace equal to its module id (`notes.*`). A module id colliding with a reserved namespace is rejected in CI.

### 3.2 Value syntax — ICU MessageFormat, restricted subset

All catalog values are ICU MessageFormat strings (via `i18next-icu`). **Only two ICU features are permitted:**

| Feature | Allowed | Example |
| ------- | ------- | ------- |
| Interpolation `{name}` | yes | `Tunggu {duration}.` |
| `plural` | yes (en, zh) | `{count, plural, one {# try left} other {# tries left}}` |
| `select` | yes | gendered/variant copy if ever needed |
| `{x, number}` / `{x, date}` / `{x, time}` ICU argument formatting | **never** | dates/numbers/money are pre-formatted by the §5 formatters and passed in as **string** params |

Rationale: ICU argument formatting depends on the engine's ICU data, which varies on Hermes; the §5 formatters are the single formatting authority.

Rules:
- Indonesian has one CLDR plural category — `id` values use plain `{count}` interpolation, never a `plural` block. `en` values use `plural` whenever a count is rendered.
- Apostrophes are ICU escape characters: use the typographic apostrophe `’` in English copy (`doesn’t`), never ASCII `'`.
- No markup/HTML in values. `<Trans>` is discouraged; prefer plain `t()`. If rich text is unavoidable, it is a design-system concern, not a catalog concern.
- Every catalog value is parsed by the ICU parser in CI; a syntax error fails the build (§7.3).

### 3.3 File layout and loading

```
packages/i18n/                       # @bolusi/i18n — universal TS, no RN imports
  src/                               # init, locale state, formatters (§5), key types
  catalogs/core/{id,en}.json         # one directory per reserved namespace (§3.1):
  catalogs/auth/{id,en}.json         #   core, auth, sync, conflict, media,
  catalogs/sync/{id,en}.json         #   push, permission, role
  catalogs/conflict/{id,en}.json
  catalogs/media/{id,en}.json
  catalogs/push/{id,en}.json
  catalogs/permission/{id,en}.json
  catalogs/role/{id,en}.json
packages/modules/notes/i18n/{id,en}.json   # each module owns its catalog files
```

- JSON files are **nested objects** (i18next `keySeparator: '.'`); the top-level module/namespace key is added at merge time, not repeated inside the file.
- The client app and the server both import `@bolusi/i18n`, which statically imports and merges every catalog under a single default namespace (`resources[locale].translation[namespace] = …`). i18next's `ns:`/`:`-separator machinery is NOT used.
- Catalogs are **bundled with the app**. No over-the-air string delivery, no remote catalog service in v0. A copy change ships as an app release.
- `react-i18next` is a dependency of the mobile app only; the server uses the bare i18next instance from `@bolusi/i18n` (§8).

### 3.4 Type-safe keys

A build script generates a TS union of all keys from the `id` catalogs (id is by definition complete, §7.1) and wires it through i18next's TS resource typing, so `t('auth.pin.typo')` is a **compile error**. Mechanism per current i18next TS docs, verified at implementation.

## 4. String discipline — zero hardcoded user-facing strings

FR-1158: no hardcoded user-facing strings anywhere, enforced from day one.

### 4.1 ESLint enforcement

`eslint-plugin-i18next`, rule `i18next/no-literal-string` at **error** severity across the app and every `screens/` directory, configured to flag:

- all JSX text nodes;
- JSX attributes in the display list: `title`, `label`, `placeholder`, `headerTitle`, `accessibilityLabel`, `accessibilityHint`, `alt`;
- callees in the display list: `Alert.alert`, `Toast.show` (extend the list when a new display API enters the codebase — extending it is part of adding the API);
- template literals in the above positions (`validateTemplate: true`).

Non-display technical literals (`testID`, style values, keys, log messages) are exempt via the rule's ignore options. Suppressing the rule with an inline disable comment requires a code-review justification — reviewers treat a bare `eslint-disable i18next/no-literal-string` as a defect.

### 4.2 DomainError mapping

Per 04-module-contract §5.1, handlers throw `DomainError(code, message)`. Discipline:

- `code` is the contract. `message` is developer-facing English for logs/diagnostics only and is **never rendered**.
- UI error rendering resolves `t('core.errors.' + code)` — the key is derived, never mapped by hand (§3.1).
- A code with no catalog entry renders `core.errors.UNEXPECTED` and logs the unknown code. Adding a new `DomainError` code therefore requires adding its `core.errors.<CODE>` row to the catalog in the same PR — CI-checked against the DomainError code registry (04-module-contract §5.2; §7.3).

### 4.3 Rejection codes

Same derivation for sync rejections (05-operation-log §8): the rejected-ops screen renders `t('core.rejection.' + rejectionCode)`. The server's `rejectionReason` string is diagnostic detail (shown in a collapsed "technical details" section, untranslated), never the primary message — silent or English-only rejection surfacing violates 05 §8.

## 5. Number, currency, date formatting

All formatting goes through `@bolusi/i18n` formatters. **UI code never calls `Intl` directly** (lint-greppable: `new Intl.` outside `packages/i18n` fails CI). All inputs are ms-epoch integers or integer IDR per repo-wide rules.

### 5.1 Money — locale-independent by design

`formatMoney(amountIdr: number): string`

- Input is **integer IDR** (no floats exist in any schema). Output is identical in every locale: `Rp` + `id-ID` digit grouping, **zero fraction digits** (FR-1160: never `Rp 250.000,00`).
- Implementation: `Intl.NumberFormat('id-ID', { style: 'currency', currency: 'IDR', minimumFractionDigits: 0, maximumFractionDigits: 0 })` — the explicit fraction-digit options are mandatory; CLDR's IDR defaults must not be trusted.
- Output is normalized: U+00A0 (NBSP, which Intl inserts after `Rp`) → regular space. Thermal printers and font fallbacks on cheap Android mangle NBSP.
- Rationale for ignoring locale: money is compared against printed receipts and between staff; one canonical rendering, no `IDR 250,000` surprise in the `en` toggle.

### 5.2 Dates, times, plain numbers

| Formatter | `id` output | `en` output | Rule |
| --------- | ----------- | ----------- | ---- |
| `formatDate(msEpoch)` | `14/07/2026` | `14/07/2026` | **Day-first in both locales** (business operates in Indonesia; US month-first ambiguity is banned) — hence `en` maps to `en-GB` (§1) |
| `formatTime(msEpoch)` | `14.30` | `14:30` | 24-hour clock, `hour12: false`; separator is whatever the Intl locale produces |
| `formatDateTime(msEpoch)` | `14/07/2026 14.30` | `14/07/2026 14:30` | date + space + time |
| `formatNumber(n)` | `1.234.567` | `1,234,567` | non-money counts/quantities; active locale grouping |

Rendering uses the device timezone; no timezone conversion logic in v0.

### 5.3 Relative time and durations — no `Intl.RelativeTimeFormat`

`Intl.RelativeTimeFormat` is unreliable on Hermes (§2) and is **never used**. Instead:

- `formatRelative(deltaMs): string` selects among catalog keys `core.time.justNow` (< 60 s), `core.time.minutesAgo`, `core.time.hoursAgo`, `core.time.daysAgo`. Staleness callers compute `deltaMs` server-relative per api/01-sync §7 — never raw device-clock arithmetic.
- `formatDuration(ms): string` (PIN-lockout waits etc.) uses `core.time.durationSeconds` / `core.time.durationMinutes`.

### 5.4 Golden vectors — must run on an Intl-capable Hermes in CI

Mirroring the RFC 8785 vector requirement (stack research; 05-operation-log §3), the following vectors run as an on-device/Hermes test in CI. A mismatch is a release blocker.

> **Which lane (measured, task 03):** these vectors are `Intl`-backed, so they need an **Intl-enabled Hermes** — the Android-emulator lane or L6 / stage 12 (task 27). They must **not** be wired into CI **stage 6** (`jcs-vectors-hermes`, 08 §5.6): that job's host VM is the standalone Hermes CLI, where `typeof Intl === 'undefined'` outright, because Intl is a Hermes **build flag** (`HERMES_ENABLE_INTL`) that RN's Android Hermes enables and the desktop CLI build does not. Running them there would produce a **false failure** on correct formatting code — and could trigger the §2 `@formatjs/intl-pluralrules` contingency on a test-VM artifact, shipping a polyfill nobody needs. Stage 6 is safe for RFC 8785 only because JCS number→string is plain ECMAScript with no Intl involved. The entry point (`packages/i18n/test/hermes-entry.ts`) is runner-free and mechanism-agnostic; whoever wires the Intl-capable lane runs it and records the outcome in `ai-docs/tasks/22-i18n.md`.

| Call | Expected |
| ---- | -------- |
| `formatMoney(250000)` | `Rp 250.000` (both locales) |
| `formatMoney(0)` | `Rp 0` |
| `formatMoney(1500500)` | `Rp 1.500.500` |
| `formatMoney(-50000)` | `-Rp 50.000` |
| `formatDate(…2026-07-14…)` | `14/07/2026` (both locales) |
| `formatTime(…14:30…)` | `14.30` (`id`) / `14:30` (`en`) |
| `formatNumber(1234567)` | `1.234.567` (`id`) / `1,234,567` (`en`) |
| `t('auth.pin.attemptsLeft', {count: 1})` in `en` | `1 try left` (proves plural selection / triggers the §2 polyfill contingency) |

## 6. Missing keys and fallback

| Situation | Behavior |
| --------- | -------- |
| Key missing in active locale (`en` or `zh`) | Render the `id` value (`fallbackLng: 'id'`); log once per key per session via `missingKeyHandler` to the client diagnostics log |
| Key missing in `id` too | Should be impossible — CI parity gates (§7.3) fail the build first. Runtime emergency degradation: `parseMissingKeyHandler` renders the humanized final key segment (`attemptsLeft` → `Attempts left`) and logs loudly. **The raw dotted key is never shown in production.** |
| Unknown error / rejection code | `core.errors.UNEXPECTED` + log (§4.2) |
| `zh` selected (V2) | fallback chain `zh → id` |

Dev builds additionally warn on every fallback hit so `en` gaps are caught while developing.

## 7. Translation workflow

### 7.1 Order and ownership

1. **`id` is written first** and is the source of truth for meaning. A feature PR that adds a key without its `id` value does not build.
2. **`en` follows in the same PR.** Nobody merges id-only keys — the parity gate (§7.3) enforces this.
3. `zh` is exempt from parity in v0 (scaffold only); a V2 agent translates the full catalog in one pass.
4. Each module's catalog files are owned by that module (§3.3); the reserved namespaces are owned by platform core. Cross-module key reuse is forbidden — shared copy belongs in `core`.
5. `ui-labels.md` is the seed and the review surface: new keys or meaning changes update that doc first (its change-control note); pure copyediting of a value may land in JSON with the doc updated in the same PR.

### 7.2 Tone register (Indonesian)

- Address the user as **`kamu`** (lowercase), never `Anda` — friendly-plain, the register of mainstream Indonesian apps; the audience is tech-inadept shop staff.
- Plain shop vocabulary, never system jargon: operations → **perubahan**, sync/upload → **terkirim / mengirim**, connectivity → **terhubung**, device → **perangkat**, store → **toko**, archive → **arsip**, photo → **foto**. Never "sinkronisasi", "operasi", "server node", "autentikasi" in user-facing copy ("server" alone is acceptable and understood).
- Short sentences. One idea per sentence. Every error states what happened and what to do next.
- English copy mirrors the same plainness; it is a working language for owners, not marketing copy.

### 7.3 CI gates (all build-blocking)

| Gate | Fails when |
| ---- | ---------- |
| Key grammar (seed doc) | any key in `ui-labels.md` violates §3.1 — checked over **every row of the doc**, whatever its namespace, and the gate fails if it parses fewer rows than the seed carries (testing-guide T-14). This is the authoritative grammar gate: the catalog-side check below only sees keys that reach a catalog, which excludes module-owned rows (`notes.*`, §3.3) — a gap that let three illegal keys ship green (task 30) |
| Key grammar (catalogs) | any key in a checked-in catalog violates §3.1 — catches a hand-edited catalog that no longer derives from the doc |
| Extraction check | any `t()` key in code is absent from the `id` catalog (and: any catalog key unused in code → warning). Derived call sites (`t('core.errors.' + code)`, §4.2/§4.3) are excluded by construction — their coverage is the error-code gate's job, not a prefix lookup |
| Parity check | `id` and `en` key sets differ (`zh` exempt in v0) |
| Collision check | same key defined by two catalogs; module id equals a reserved namespace |
| ICU syntax check | any catalog value fails the ICU MessageFormat parser, or uses a forbidden ICU feature (§3.2) |
| Error-code coverage | any code in the DomainError registry (04-module-contract §5.2) or 05 §8 rejection code lacks its `core.errors.*` / `core.rejection.*` row |
| Hermes golden vectors | §5.4 mismatch on device |
| `no-literal-string` | §4.1 |

## 8. Server-side use (push notifications)

Push notification title/body are composed **server-side** (expo-notifications payloads must carry final display text for killed-app delivery). The server imports the same `@bolusi/i18n` package and renders the `push.*` templates (ui-labels.md) with the **recipient user's last-synced locale preference** (§1.1 `user_prefs` projection, server side; fallback `id`). One catalog, two runtimes — there is no second string source. Transport, categories, payload rules, and routing are owned by `api/04-push.md`, not this doc.

## 9. What i18n is NOT

- **Not OTA content delivery** — catalogs ship in the app bundle (§3.3).
- **Not per-tenant customization** — every tenant sees the same strings in v0; white-labeling is a post-SaaS concern (roadmap).
- **Not translation of user data** — note bodies, names, and payload contents render verbatim.
- **Not locale negotiation** — the server never inspects `Accept-Language`; locale is the synced user preference only.
