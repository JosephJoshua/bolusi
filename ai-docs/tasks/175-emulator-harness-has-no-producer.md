## Root cause — the producer does not exist

`harness-device.mjs` launches `com.bolusi.app/com.bolusi.app.HarnessActivity` with `--es bolusiHarnessRunId <id>` (`:167–177`), then polls `adb logcat -d -s BOLUSI_HARNESS_RESULT:I` every 5 s looking for the substring `BOLUSI_HARNESS_RESULT:` (`extractResultPayload`, `:47`). **Every link on the far side of that `am start` is missing.** Four independent, statically-verifiable defects — none needs a device to confirm:

### A. There is no `HarnessActivity`

`grep -rn "HarnessActivity"` over the whole repo (node_modules excluded) returns **exactly one hit**:

```
scripts/harness-device.mjs:150:  const activity = 'com.bolusi.app/com.bolusi.app.HarnessActivity';
```

— the driver's own string literal. `apps/mobile/android/` is **not tracked** (generated by `expo prebuild`); `app.config.ts`'s `plugins` list holds only stock Expo plugins; the repo contains **no** `withAndroidManifest` config plugin. There is therefore no mechanism by which the generated manifest could declare that component. The driver has been aiming at a component that has never existed.

**And the launch check is a guard that is green for the wrong reason (§2.11).** `harness-device.mjs:178` gates only on `launch.status !== 0`, and the log proves that returned **0** for a component that provably does not exist. `am start` reports component-not-found as `Error type 3` / `Error: Activity class {…} does not exist.` on **stdout** while exiting 0 — and `launch.stdout` is never read. The one check that existed to catch exactly this could not.

### B. Nothing anywhere emits the tag — T-16, a mention is not a producer

Exhaustive search across `*.ts,*.tsx,*.mjs,*.js,*.kt,*.java` for `BOLUSI_HARNESS_RESULT` / `HARNESS_RESULT_TAG` / `bolusi-harness-result`. Every hit is a **definition, an assertion about the definition, or the consumer**:

| site | what it is |
| ---- | ---------- |
| `apps/mobile/src/harness/flag.ts:19` | the constant's definition |
| `apps/mobile/src/harness/flag.test.ts:25` | asserts the constant equals its own literal |
| `apps/mobile/src/harness/result.ts` | the *type* of the document (no writer) |
| `scripts/harness-device.mjs:17,47,184` | the consumer + its logcat filter |
| `packages/test-support/src/harness-device.test.ts` | the consumer's unit-test fixtures |

There is **no `console.log`, no `android.util.Log.i`, no writer of any kind.** The document type is declared, the tag is declared, the parser is tested against synthetic fixtures — and nothing ever produces one.

### C. `loadHarness()` has zero production callers

`grep -rn "loadHarness"` → the definition (`registry.ts:49`), three comments, and `flag.test.ts` (its own test). Same shape for `runAtRestGate`: `registry.ts` + its own test. `apps/mobile/src/harness/part-c/` contains only `at-rest-device-ctx.ts` and `budgets.ts` — **no result emitter, no harness screen.**

This is precisely the class CLAUDE.md §2.11 already names: *"a function with 11 sound tests and zero callers, whose green stays green when the line that actually gates the screen is broken."*

Task 27's own **Files / modules touched** list names what is missing, in its own words:

> `apps/mobile/src/harness/part-c/` — gate runners P-1…P-6 + write-throughput benchmark, budget constants, **result JSON emitter**

…and its Goal opens with *"the hidden Harness screen's Part C section in `apps/mobile`"*. The screen and the emitter were never built.

**Related, same cause:** 4 of the 7 gates the driver *requires* have no runner at all. `registry.ts` wires only `runAtRest`; `SEC-AUTH-09-leg1`, `SEC-OPLOG-06-jcs`, `CHAOS-01/03/06/07` exist solely as ids in `EMULATOR_CORRECTNESS_GATE_IDS` under a comment asserting they "run through the shared engines … on device". No code does. Even a working emitter would fail on `missing gate …` until these are wired.

### D. Second, independent defect — `BOLUSI_TEST_HARNESS` cannot reach a release bundle

`harnessEnabled()` (`apps/mobile/src/harness/flag.ts:14`) is:

```ts
return process.env['BOLUSI_TEST_HARNESS'] === '1';
```

Expo inlines **only `EXPO_PUBLIC_`-prefixed** variables into the client bundle (Expo docs, *Environment variables* / *EAS environment variables*: "Use the `EXPO_PUBLIC_` prefix to make variables available in your application code via `process.env`"; `EXPO_NO_CLIENT_ENV_VARS=1` disables even that). `BOLUSI_TEST_HARNESS` carries no prefix, so it is **not** inlined — at runtime in the release bundle the expression is `undefined === '1'` → `false`, and `loadHarness()` returns `null`.

The contrast is visible in one env block, `.github/workflows/ci.yml:511–513`:

```yaml
        env:
          BOLUSI_TEST_HARNESS: '1'          # ← line 512, NOT inlined into the bundle
          EXPO_PUBLIC_API_URL: '…'          # ← line 513, inlined
```

`apps/mobile/eas.json:25` sets the same unprefixed name for the EAS `test` profile, with the same consequence. **So even after A–C are fixed, the harness stays shut.** `flag.ts:3`'s comment — *"the env the EAS `test` profile sets"* — is accurate about the build shell and wrong about the bundle: a comment is a hypothesis, not evidence (§2.11), and "typed and compiling" is not "running on the target". `harness-flag.test.ts` and `eas-profiles.test.ts` assert the flag's presence in **`eas.json`** and `flag.test.ts` mutates `process.env` **in Node** — no test has ever asked whether the value survives Metro.

## Why no crash evidence exists, and what that means for task 148

**The lane emits zero logcat, by construction.** `sh()` (`harness-device.mjs:143–145`) calls `spawnSync` **without** `stdio: 'inherit'`, so every byte of adb stdout/stderr is captured into a buffer and discarded. The log confirms it: between `08:40:00.3550736` (blank line after the pnpm banner) and `09:00:13.7138622` (the failure) there is **not one line of output** — no `Performing Streamed Install`, no `Starting: Intent`, no `Status: ok`, nothing. And the poll filters logcat to `-s BOLUSI_HARNESS_RESULT:I`, a **tag** filterspec that excludes `AndroidRuntime`, `ActivityManager` and every crash tag by design.

So: `FATAL EXCEPTION`, `Process … has died`, ANR and `Force finishing activity` were all searched for and **none appears — but none could have.** Absence here is not evidence.

**Task 148 is therefore neither implicated nor exonerated by this run, and it is not needed to explain the failure.** The missing producer is a static fact provable from the repo without a device, and it accounts for the symptom completely. Consistent with the rest of the board: `ios-simulator` is green and every Node lane passes, because none of them exercises this path. **Do not spend a re-run investigating 148 until A–D are fixed** — but see the observability step below, which answers the 148 question for free.

## Deliverable

Not a one-line fix: this is 27a's unbuilt half. Land it in this order, each leg falsified before it is believed (§2.11).

1. **Make the flag reachable at runtime.** Rename to `EXPO_PUBLIC_BOLUSI_TEST_HARNESS` (or thread it via `app.config.ts` → `extra` → `expo-constants`, which is prefix-independent and arguably cleaner for a non-public flag). Touches `flag.ts:14`, `ci.yml:512`, `eas.json:25`, `harness-flag.test.ts`, `eas-profiles.test.ts`, `flag.test.ts`. **Falsify:** grep the built release bundle (`apps/mobile/android/app/build/…/index.android.bundle`) for the literal, and confirm the *old* name is absent from it — that is the observation nobody has ever made. Keep the "never in production/preview" static check intact under the new name.

2. **DECIDE THE TAG AND THE FILTER TOGETHER — there is a trap here.** A React Native `console.log` reaches logcat under tag **`ReactNativeJS`**, *not* `BOLUSI_HARNESS_RESULT`. `extractResultPayload` greps for the substring `BOLUSI_HARNESS_RESULT:` anywhere in a line, so `console.log('BOLUSI_HARNESS_RESULT: {…}')` **would** match textually — but `harness-device.mjs:184`'s `-s BOLUSI_HARNESS_RESULT:I` filters by **tag**, so that very line is filtered OUT before the parser ever sees it. Either emit through a native module that calls `android.util.Log.i(HARNESS_RESULT_TAG, json)`, or drop `-s` and let the substring grep do the work on an unfiltered dump. Picking one in isolation reproduces this exact failure with a producer in place.

3. **Build the emitter + entry point.** A path that, with the flag on, calls `loadHarness()`, runs the 7 `EMULATOR_CORRECTNESS_GATE_IDS`, and writes exactly one result document. Wire the 4 gates that currently have ids but no runner (B, "related"). For the entry point, choose one and delete the other from the driver:
   - **(i)** declare a real `HarnessActivity` via a `withAndroidManifest` config plugin (matches the driver as written; most native machinery), or
   - **(ii)** drop the separate activity, launch `MainActivity`, and let the JS root branch on the flag. The run id then needs a JS-readable channel — an `--es` intent extra is *not* readable from RN without a native module, so prefer a deep link (`am start -a android.intent.action.VIEW -d 'bolusi://harness?runId=…'` + `Linking.getInitialURL()`). Least native code; **verify the scheme is registered in the generated manifest** before relying on it.

4. **Make the launch check load-bearing (A).** `am start` exits 0 on a missing component, so assert on `launch.stdout`: fail unless it contains `Status: ok` / contains no `Error:`. **Falsify:** point the driver at a deliberately bogus component and watch it go red in under a second instead of hanging 20 minutes.

5. **Observability — do this even if it ships alone.** Stop swallowing adb output: on any failure path, print `launch.stdout`/`stderr` and dump `adb logcat -d` (unfiltered, bounded tail) into the job log and into `maestro-artifacts/`. Right now a 20-minute red run produces **one line** of diagnosis.
