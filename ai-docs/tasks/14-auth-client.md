# TASK 14 — auth-client (enrollment, device keys, offline PIN + lockout, switcher, idle lock)

**Status:** todo
**Depends on:** 03, 04, 09, 10, 13

## Goal

Deliver the client half of auth per api/02-auth §4/§6 as a platform-free auth runtime in `@bolusi/core` plus the two `apps/mobile` port implementations it needs. Slices: enrollment wizard **flow logic** (owner login → store pick → Ed25519 keypair gen via quick-crypto → `POST /v1/devices/enroll` → bundle persisted to the client directory tables BEFORE the genesis op is appended); `KeyStorePort` impl on expo-secure-store (`bolusi.device_private_key`, `bolusi.device_token`; values < 2 KB, qualified claims per D10/security-guide); offline PIN verify (async argon2id via quick-crypto against `user_pin_verifiers`, greatest-`asOf` effective-verifier merge, constant-time compare); the `PinAuth` lockout machine exactly per api/02-auth §6.5 (persisted `pin_attempt_state`, `PIN_RATE_LIMITED`/`PIN_LOCKED` without running the KDF, owner unlock `auth.clearPinLockout` + reset recovery, client-side main_owner-target rule); user-switcher session state with `auth.user_switched`/`auth.session_ended` runtime emissions; idle lock that preserves per-user in-progress work; and PIN first-set/change/reset flows emitting `auth.pin_changed`/`auth.pin_reset` plus the queued verifier `POST /v1/users/:userId/pin-verifier` on next online contact. Everything is exercisable headlessly on Node via injected ports (noble crypto, better-sqlite3 Kysely, FakeClock). Auth **screens** are task 24; device-revocation wipe handling rides sync (task 15); the server push-validation arms live in tasks 13/16 — none of that here.

- **Enrollment flow tests:** step order per api/02-auth §4.1 asserted — private seed in KeyStore before the enroll POST; on `EnrollRes`: token → KeyStore, bundle → all four directory tables (verifiers for own store only) — and the genesis `auth.device_enrolled` (seq 1, evaluator-exempt, userId = enrolling owner) appended **only after** directory persist. **Bundle-before-command bootstrap test:** the very first command after enrollment gets an evaluator ALLOW/DENY answered from directory rows (02-permissions §6 bootstrap rule). Crash-retry test: process dies between enroll response and token persist → retry reuses the same `Idempotency-Key`. Enrollment logs nobody in (post-enroll state = switcher, no open session).
- **Verifier + PIN verify tests:** SEC-AUTH-01 — verifier construction rejects out-of-bounds params (`mKiB < 19456` or `> 65536`, `t ∉ [2,4]`, `p ≠ 1`, salt ≠ 16 B, hash ≠ 32 B); params record round-trips into verification (self-describing, never guessed). Merge-rule units: local row newer → local wins; bundle newer → bundle wins; nil-device control-plane `asOf` loses to a real op position at equal timestamp; equal triples are impossible by construction (canonical order total). Constant-time compare (`timingSafeEqual`-equivalent) and verifier rows existing only in the SQLCipher DB are asserted here as the client precursors of SEC-AUTH-09 (full storage-scan/push-scan lands with 26/28; SEC-AUTH-10 benchmark is task 27).
- **Lockout machine (SEC-AUTH-02, -03, -04, -05):** code-table parity test against 03-state-machines §9.2 (states derived from the persisted row, not stored); FakeClock-exact schedule — attempt at `delay − 1 ms` → `DomainError('PIN_RATE_LIMITED', {retryAt})`, at `delay` → evaluated; KDF-invocation spy proves attempts during delay/lockout never run argon2id (wrong OR correct PIN); 10th consecutive failure → `locked_out`, emits `auth.pin_locked_out` `{consecutiveFailures, windowStartedAt}` (`source: 'system'`, userId = target); success resets counter to 0; per-(userId, deviceId) isolation — user B unaffected by A's failures; restart persistence (drop in-memory state, reopen DB → counter + `notBefore` intact); clock rollback (FakeClock −1 h → stored `notBefore` stands, never recomputed downward); both recovery paths asserted **offline** (transport fake counts zero calls): `auth.clearPinLockout` by an `auth.pin_unlock` holder emits `auth.pin_lockout_cleared` + resets, and a newer-`asOf` verifier (local reset or bundle refresh) clears lockout as an auth-runtime side effect — not an applier. Invalid transitions → `INVALID_TRANSITION`. No online self-recovery code path exists (grep-level assertion is acceptable).
- **Command permission/targeting denials (SEC-AUTH-06 client arm, SEC-AUTH-11 client arm):** `auth.changePin` targeting non-self → `PERMISSION_DENIED` reason `restriction_violated` (02-permissions §5.4.6); `auth.resetPin` without `auth.user_reset_pin` → `PERMISSION_DENIED` + `auth.permission_denied` emitted; reset/unlock target absent from `users_directory` → denied; store_owner actor resetting a main_owner-role holder → denied at the command layer; main_owner actor on same target → allowed; `auth.clearPinLockout` without `auth.pin_unlock` → denied. (Forged-op push rejection `SCOPE_VIOLATION` is the server arm — tasks 13/16; cross-reference in test names, do not implement here.)
- **PIN flow tests:** first-PIN (bundle `pinVerifier: null` → forced setup, current-PIN check skipped only in this case, fresh 16-byte salt, local `user_pin_verifiers` write with `asOf` = emitting op's canonical position); change (current-PIN local verify precedes execution); reset (verifier computed on the resetting device, applied immediately, target's lockout cleared). Each emits its op with payload exactly `{targetUserId, verifierRef}` — schema test proves no salt/hash fields exist, plus a serialized-payload byte scan across a full change/reset cycle finds no salt/hash material. Verifier POST queue: sent on next online contact with `verifierRef` equal to the op's; `applied: false` (stale) response is terminal-idempotent — no retry loop, no local rollback. Idempotency: replaying a queued POST after a crash converges (greatest-`asOf`).
- **Switcher/session tests (SEC-AUTH-07):** switch A→B after B's PIN verify emits `auth.session_ended` (`reason: 'switch'`, payload = A's session) then `auth.user_switched`, both with envelope userId = B; ops appended after the switch carry userId = B, before carry A, and the device chain continues unbroken; switching-in requires no permission; deactivated users are not switcher-usable while their names remain resolvable (03-state-machines §6); a user switch invalidates the permission memo (02-permissions §6 hook invoked).
- **Idle lock (SEC-AUTH-08, runtime level):** FakeClock past `idleLockSeconds` (bundle value; clamp 60–3600, default 300 honored) → `auth.session_ended` (`reason: 'idle_lock'`, `source: 'system'`); manual lock → `reason: 'manual_lock'`, `source: 'ui'`; per-user work state keyed by userId survives the lock and is restored on that user's next unlock, and is NOT restored to a different user.
