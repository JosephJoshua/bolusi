## ADDED 2026-07-22 from the task-128 review

8. **MEDIUM — `packages/ui/src/components/TextInput.tsx` permits `multiline` + `secureTextEntry` together, which RN documents as broken.** The RN 0.86 docs say verbatim: *"`secureTextEntry` — If `true`, the text input obscures the text entered… **Does not work with `multiline={true}`.**"* Both props are independent optional booleans (`:47,55,116,118`), so `<TextInput secureTextEntry multiline />` type-checks and yields a **silently unmasked password field**.
   - **Not a regression** — the reviewer confirmed three ways that the enrollment password field cannot receive it today: `EnrollmentScreen.tsx` is not in task 128's diff at all, `:133` passes `secureTextEntry` with no `multiline`, and a new call-site test asserts both `secureTextEntry === true` and `multiline === false` on that node (watched red under falsification).
   - **But task 128 created the footgun**, in a contended package fronting the enrollment password, and the only thing standing in the way is a test enumerating *today's* call sites — which 128's own falsification B proved cannot see a new one (removing `multiline` from the call site left the entire `ui` suite green). The component's header comment is otherwise exhaustive about multiline caveats and omits this one.
   - **Fix by construction (CLAUDE.md §2.5):** `multiline={multiline && !secureTextEntry}`, or a discriminated props union that makes the combination unrepresentable. A test enumerating call sites is the weaker answer and is what this repo keeps getting caught by.
9. **LOW — `packages/ui/src/tokens.ts:193-195` justifies the multiline height cap with a mechanism that isn't the mechanism.** It says the cap ensures the field "can never push the §8.1 bottom action bar off a 360x640 screen"; `AppShell.tsx:72` styles content `{flex:1}` with `bottomAction` as a sibling of intrinsic height, so the bar cannot be pushed off regardless. **The value is right; the stated reason is not the reason.** The real residual, which the reviewer could not test in any available lane: there is no `ScrollView` or `KeyboardAvoidingView` anywhere in `packages/ui/src/shell` or `packages/modules/src/notes`, so a 128 dp minimum inside a non-scrolling `flex:1` View could squeeze the create-mode attach row out of reach with the keyboard open. Pre-existing, mildly amplified by 128, unverified on target — and now doubly blocked, since **task 148** means no Android APK exists to check it on.
10. **LOW — `apps/mobile/e2e-web/visual.spec.ts:214,218` hard-code `26` for `type.body.lineHeight`.** If the type scale moves, the wrap threshold silently drifts instead of failing. Import the token.
