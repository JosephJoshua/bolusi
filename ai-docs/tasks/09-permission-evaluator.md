# TASK 09 — Permission evaluator (@bolusi/core authz)
**Status:** todo
**Depends on:** 02, 04
## Goal
Deliver the authorization subsystem of `@bolusi/core` per 02-permissions §5: the static permission registry assembled at startup from module-manifest `permissions` blocks, with **startup failure** on duplicate id, id-prefix/module mismatch, or any command/query permission reference that does not resolve (02 §3.2 rules 1–4). Implement the normative `hasPermission` algorithm (02 §5.2) — tenant vs store scope, grant union, malformed-grant drop, unconditional fail-closed with the exact `DenialReason` set — evaluating client-side against the directory mirror tables (`users_directory`, `roles_directory`, `user_roles_directory`, 10-db §9.5) through the injected Kysely handle, synchronously via the per-`(userId, storeId)` memoized effective-set snapshot with **event-driven-only** invalidation (bundle-refresh write, user switch — never TTL; 02 §6). Implement the denial emitter: builds `auth.permission_denied` payloads (02 §7 shape, all keys always present), throttles to one op per `(userId, permissionId, target)` per 5-minute window with in-memory `suppressedRepeats`, and emits through an injected runtime hook/port — actual op append is task 06's, and `ctx.requirePermission` wiring is task 10's; this task exposes evaluator + emitter + hooks as ports the command runtime consumes. Also deliver the permission-set-change invalidation hook registry for module-declared caches of privileged data (02 §8.4, grow and shrink both firing per §8.5). No UI, no server endpoints, no bundle-fetch logic (that is task 14) — this slice is pure `@bolusi/core` + tests.

## Acceptance
- **Done-condition:** `@bolusi/core` test suite green on Node CI (L1 + L2), typecheck and `bolusi/boundaries` lint clean — core stays platform-free (08-stack-and-repo §3.3: imports only `schemas` + kysely types; clock via `ClockPort`, no `Date.now()`, no timers).
- **Registry assembly tests:** multi-module fixture merges correctly; duplicate id across two modules → startup failure (not a warning); entry whose id prefix ≠ declaring module id → startup failure; a command/query referencing a permission absent from the registry → **startup failure** (the named startup-failure test from the brief); a role's grant list containing an id unknown to this build is inert — contributes nothing, rest of the role unaffected (02 §3.2).
- **Fail-closed matrix — one test per row of 02 §5.3, asserting the exact `DenialReason`:** unknown permission id → `unknown_permission`; user absent from `users_directory` → `user_inactive`; `status = 'deactivated'` → `user_inactive`; user tenant ≠ evaluation tenant → `tenant_mismatch`; store-scoped permission with null/absent `storeId` → `missing_scope`; grant referencing a deleted/unknown role ignored → `not_granted` (dangling-role test); no matching grant → `not_granted`; evaluator forced to throw (corrupt `permission_ids` JSON row) → `evaluation_error`, never an exception to the caller.
- **Scope cases (02 §5.2 step 4, fixtures from §10/§12 roles):** tenant-scoped permission satisfied only by a `storeId = null` grant — a store-scoped grant of a role that *contains* the id still denies; store-scoped permission satisfied by a matching-store grant AND by a tenant-wide grant (main_owner allowed in every store); grant for a different store denies; union semantics — two roles each holding half the ids together allow both (FR-1023); store-scoped role granted with null `storeId` is dropped as malformed; `scope: 'tenant'` ignores the passed `storeId`. Include the 02 §12 built-in denial fixtures: `staff` attempting `auth.user_create`, zero-grant user attempting `notes.create`.
- **Memo tests:** evaluation is synchronous; with no invalidation event, a direct directory-row change is NOT observed (snapshot proves memoization); after a simulated bundle-refresh write → recompute observed; after user switch → recompute; no TTL/timer exists in the module (FakeClock advances alone never invalidate).
- **Denial emitter tests (FakeClock, testing-guide §3.3):** first denial emits exactly one payload `{ permissionId, surface, target, reason, scopeStoreId, suppressedRepeats: 0 }` — all keys present, `scopeStoreId` null for tenant-scope checks; repeats inside the 5-min window emit nothing and count; first emission after the window carries `suppressedRepeats = N`; distinct `(userId, permissionId, target)` tuples throttle independently; counter is memory-only (reset on simulated restart, accepted); emission itself is never permission-checked (no recursion — assert evaluator not re-entered); payload validates against the `@bolusi/schemas` shape.
- **Invalidation-hook tests:** a hook registered for permission ids fires when a permission-set change touches those ids, on both grow and shrink (02 §8.5); unrelated changes do not fire it.
- **SEC-\*/CHAOS-\*:** no `SEC-*` id in security-guide §12 is owned by this surface — SEC-AUTH-06/-11's local-denial legs *consume* this evaluator and land with tasks 13/14/28; CHAOS-11 (PIN) belongs to task 14. The §5.3 fail-closed matrix above IS this surface's adversarial suite and ships in this task, before review (CLAUDE.md §2.5). Do not invent new SEC ids (security-guide §2 owns the namespace).
- **Lint/CI gates:** suites wired into the every-PR Node pipeline; deterministic (seeded fixtures, FakeClock, no wall-clock waits — testing-guide §1); the 02 §2 CI lint (manifest declares only own-prefix permissions; command/query requires only own-module permissions) is verified present in `tooling/eslint` — if task 01 did not ship it, add the rule + failing fixture here, since this task implements its runtime backstop.
